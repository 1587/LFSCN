iii. LFS 目标架构

LFS 基本的目标架构是 AMD/Intel x86 (32 位) 和 x86_64 (64 位) CPU。另一方面，本书中的指令已知可以经过修改后用于 PowerPC CPU。要编制一个使用这些 CPU 之一的系统，主要的前提条件比此后几页提到的多一项，就是一个已存在的 Linux 系统，比如更早安装的 LFS、Ubuntu、Red Hat、Fedora、SuSE 或者其他指定对应架构的发行版。此外注意，32 位发行版可以安装于 64 位 AMD/Intel CPU 的电脑上并用作宿主系统。

关于 64 位系统还有一些事情要提。与 32 位系统相比，64 位可执行文件的尺寸略大一点，执行速率也只是略快一点。举个例子，某次在基于 Core2Duo CPU 的 LFS-6.5 编制测试中，测得了以下数据：

  • 32 位架构编制时间为 198.5 分钟，编制尺寸为 648 MB。

  • 64 位架构编制时间为 190.6 分钟，编制尺寸为 709 MB。

不难发现，64 位编制只比 32 位快了 4% 却大了 9%。寻求 64 位系统的好处相对不大。当然，如果你有超过 4GB 的 RAM 或者想要修改超过 4GB 的数据，64 位系统还是很有优势的。(译注：32 位操作系统能够使用的内存无法超过 4 GB)

LFS 的默认 64 位编制产生的是 “纯粹的” 64 位系统。也就是说，它只支持 64 位可执行文件。编制 “多重库的” 系统需要编制很多应用程序两次，32 位和 64 位各一次。LFS 并不直接支持这种方式，因为那会干扰我们提供编制简单基础的 Linux 系统所需指导的教学目标。关于这个更高级的话题可以参考 CLFS 工程。

关于 64 位系统还有最后一点要说，有些程序包当前无法在 “纯粹的” 64 位系统上编制或者需要特定的编制指令。通常这些程序包中嵌入了 32 位特有的汇编指令，在 64 位系统上编制会失败。这样的程序包括 此处 的一些传统图形卡的 Xorg 驱动。这些问题多数可以解决，但可能需要特定的过程或补丁。
