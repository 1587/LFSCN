5.5. GCC-4.7.2 - 第一遍

GCC 程序包包含 GNU 编译器集，其中包括 C 和 C++ 编译器。

预计编制时间：5.5 SBU
需求磁盘空间：1.4 GB

5.5.1. 安装交叉 GCC

GCC 现在需求 GMP、MPFR 和 MPC 程序包。由于这些程序包可能未包含与你的宿主发行版中，它们将与 GCC 一同编制。将每个程序包解包至 GCC 源代码目录并重命名产生的文件夹使得 GCC 编制过程能够自动使用他们：

注意

关于这一节经常出现误解。此过程与其他每一节都是一样的，就像之前在程序包编制指令中解释的那样。首先从源代码目录中提取 gcc 压缩包然后改变到创建的目录，到这个时候再继续运行以下指令。 

tar -Jxf ../mpfr-3.1.1.tar.xz
mv -v mpfr-3.1.1 mpfr
tar -Jxf ../gmp-5.0.5.tar.xz
mv -v gmp-5.0.5 gmp
tar -zxf ../mpc-1.0.1.tar.gz
mv -v mpc-1.0.1 mpc

以下命令将改变 GCC 的默认动态链接器位置以使用安装于 /tools 的副本。它也会从 GCC 的包含搜索路径中移除 /usr/include。执行： 

for file in \
 $(find gcc/config -name linux64.h -o -name linux.h -o -name sysv4.h)
do
  cp -uv $file{,.orig}
  sed -e 's@/lib\(64\)\?\(32\)\?/ld@/tools&@g' \
      -e 's@/usr@/tools@g' $file.orig > $file
  echo '
#undef STANDARD_STARTFILE_PREFIX_1
#undef STANDARD_STARTFILE_PREFIX_2
#define STANDARD_STARTFILE_PREFIX_1 "/tools/lib/"
#define STANDARD_STARTFILE_PREFIX_2 ""' >> $file
  touch $file.orig
done

由于上文似乎很难跟进，我们把它拆解开来。首先我们找到 gcc/config 目录中所有名为 linux.h、linux64.h 或者 sysv4.h 的文件。对于找到的每个文件，我们把它复制为一个有同样名字但是加上 “.orig” 后缀。然后第一个 sed 表达式将出现的所有 “/lib/ld”、“/lib64/ld” 或 “/lib32/ld” 替换为 “/tools”，第二个替换不可更改的 “/usr”。然后我们在文件末尾添加我们的声明，调节包含搜索路径和起始文件前缀。最后我们用 touch 来更新复制的文件的时间戳。在与 cp -u 联用时这可以防止由于误操作将命令运行两次而改变原有文件。

GCC 不能正确检测堆栈保护，会导致 Glibc-2.16.0 编制时出现问题。应用以下命令修复之： 

sed -i '/k prot/agcc_cv_libc_provides_ssp=yes' gcc/configure

GCC 文档推荐在源代码目录外创建目录编制 GCC： 

mkdir -v ../gcc-build
cd ../gcc-build

准备编译 GCC： 

../gcc-4.7.2/configure         \
    --target=$LFS_TGT          \
    --prefix=/tools            \
    --with-sysroot=$LFS        \
    --with-newlib              \
    --without-headers          \
    --with-local-prefix=/tools \
    --with-native-system-header-dir=/tools/include \
    --disable-nls              \
    --disable-shared           \
    --disable-multilib         \
    --disable-decimal-float    \
    --disable-threads          \
    --disable-libmudflap       \
    --disable-libssp           \
    --disable-libgomp          \
    --disable-libquadmath      \
    --enable-languages=c       \
    --with-mpfr-include=$(pwd)/../gcc-4.7.2/mpfr/src \
    --with-mpfr-lib=$(pwd)/mpfr/src/.libs

配置选项的含义：

--with-newlib

由于现在还没有能够工作的 C 库可用，此项确保编制 libgcc 时 inhibit_libc 常量是已经定义了的。此项防止编译任何需要 libc 支持的代码。

--without-headers

在创建一个完整的交叉编译器的时候，GCC 需要兼容于目标系统的标准头文件。出于我们的目的，不需要这些头文件。此开关防止 GCC 寻找它们。

--with-local-prefix=/tools

本地前缀是系统中 GCC 搜索本地安装的包含文件的位置。默认值是 /usr/local。设置它为 /tools 帮助保持宿主上的 /usr/local 在 GCC 的搜索路径之外。

--with-native-system-header-dir=/tools/include

GCC 默认搜索 /usr/include 寻找系统头文件。与 sysroot 开关联用时，它通常会被翻译为 $LFS/usr/include。然而下两节将要安装的头文件会去往 $LFS/tools/include。此开关确保 gcc 会正确地找到他们。在第二遍的 GCC 中，相同的开关将会确保不会找到宿主上的头文件。

--disable-shared

此开关强制 GCC 静态链接内部库以防止可能出现的有关宿主系统的问题。

--disable-decimal-float, --disable-threads, --disable-libmudflap, --disable-libssp, --disable-libgomp, --disable-libquadmath

这些开关禁用十进制浮点扩展、线程、libmudflap、libssp 和 libgomp 和 libquadmath 的支持。这些特性在编制交叉编译器时会编译失败，而且对于交叉编译临时的 libc 没有必要。

--disable-multilib

在 x86_64 上，LFS 还不支持多重库配置。此开关对 x86 是无害的。

--enable-languages=c

此选项确保只编制 C 编译器。这是目前需要的唯一的语言。

运行以下命令编译 GCC： 

make

编译现在完成了。通常我们现在要运行测试套件，但现阶段测试套件框架 (Tcl、Expect 和 DejaGNU) 还没有到位。在这里运行测试意义不大，因为这第一遍编译的程序很快会被第二遍的替换掉。

安装程序包： 

make install

使用 --disable-shared 意味着 libgcc_eh.a 文件不会创建和安装。Glibc 程序包因为在自己的编制系统中使用了 -lgcc_eh 而依赖这个库。此项依赖关系可以通过创建一个到 libgcc.a 的符号链接来满足，因为这个文件最后会包含通常包含在 libgcc_eh.a 中的对象。 

ln -sv libgcc.a `$LFS_TGT-gcc -print-libgcc-file-name | sed 's/libgcc/&_eh/'`

有关此程序包的细节位于第 6.17.2 节 “GCC 的内容”。
