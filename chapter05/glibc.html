5.7. Glibc-2.16.0

Glibc 程序包包含主要的 C 库。这个库提供用于分配内存、搜索目录、开关读写文件、字符串处理、表达式匹配和算数等的基本途径。

预计编制时间：5.4 SBU
需求磁盘空间：554 MB

5.7.1. 安装 Glibc

在某些情况下，尤其是在 LFS 7.1 中，rpc 头文件并没有正确安装。检测它们在宿主系统上是否已经安装，如果没有则安装之： 

if [ ! -r /usr/include/rpc/types.h ]; then 
  su -c 'mkdir -p /usr/include/rpc'
  su -c 'cp -v sunrpc/rpc/*.h /usr/include/rpc'
fi

修复一个以 GCC-4.7.2 编制 Glibc-2.16.0 时出现的问题： 

sed -i 's/ -lgcc_s//' Makeconfig

Glibc 文档推荐在源代码目录外创建目录编译 Glibc： 

mkdir -v ../glibc-build
cd ../glibc-build

接下来，准备编译 Glibc： 

../glibc-2.16.0/configure                             \
      --prefix=/tools                                 \
      --host=$LFS_TGT                                 \
      --build=$(../glibc-2.16.0/scripts/config.guess) \
      --disable-profile                               \
      --enable-add-ons                                \
      --enable-kernel=2.6.25                          \
      --with-headers=/tools/include                   \
      libc_cv_forced_unwind=yes                       \
      libc_cv_ctors_header=yes                        \
      libc_cv_c_cleanup=yes

配置选项的含义：

--host=$LFS_TGT, --build=$(../glibc-2.16.0/scripts/config.guess)

这几个参数告诉 Glibc 使用 /tools 中的编译器和链接器进行交叉编译。

--disable-profile

不使用检测的信息进行编译。当需要检测信息时，去掉这个参数。

--enable-add-ons

告诉 Glibc 使用附加的 NPTL 作为线程库。

--enable-kernel=2.6.25

告诉 Glibc 提供 2.6.25 或更高的内核的支持。比这个旧的内核将不会提供支持。

--with-headers=/tools/include

告诉 Glibc 使用 tools 文件夹里的头文件编译自身，这样可以让 Glibc 根据新的内核进行优化。

libc_cv_forced_unwind=yes

在 第 5.4 节 “Binutils-2.23.1 - 第一遍” 安装的链接器是交叉编译得到的，在 Glibc 安装好之前，它不能正常工作。这意味着 configure 对 force-unwind 的检查会失败。libc_cv_forced_unwind=yes 告诉 configure 脚本不进行这项检查。

libc_cv_c_cleanup=yes

同样，我们传递 libc_cv_c_cleanup=yes 给 configure 脚本跳过检查但仍配置 C 清理控制支持。

libc_cv_ctors_header=yes

同样，我们传递 libc_cv_ctors_header=yes 给 configure 脚本跳过检查但仍配置 GCC 构造函数支持。

在这步进行时，可能会有下面的警告： 

configure: WARNING:
*** These auxiliary programs are missing or
*** incompatible versions: msgfmt
*** some features will be disabled.
*** Check the INSTALL file for required versions.

这个丢失或不合适的 msgfmt 程序在这里是无害的。这个 msgfmt 是 Gettext 的一部分，宿主系统应该会提供。

编译程序包： 

make

此程序包包含测试套件，但现在还不能运行，因为我们还没有 C++ 编译器。

注意

这些测试的成功执行同样需要正确安装的区域数据。区域数据提供系统默认的信息，比如日期、时间、货币格式以及系统工具的输出。如果在本章不运行测试套件，就没必要现在安装区域数据 (推荐)。合适的区域数据将会在下一章安装。如果无论如何都要安装 Glibc 区域数据，参考第 6.9 节 “Glibc-2.16.0”。

安装程序包： 

make install

小心

在这里，停下来确保新工具链的基本功能 (编译和链接) 能正常工作很重要。要进行稳定性检查，运行以下命令： 

echo 'main(){}' > dummy.c
$LFS_TGT-gcc dummy.c
readelf -l a.out | grep ': /tools'

如果一切工作正常，就不会出现错误，最后一条命令的输出会符合以下格式： 

[Requesting program interpreter: /tools/lib/ld-linux.so.2]

留意 /tools/lib 或者 64 位计算机上的 /tools/lib64 作为动态链接器的前缀出现。

如果输出不如上文所示或者根本没有输出，那就有地方出错了。研究回溯刚才的步骤，查出问题并修正之。在继续之前必须解决这一问题。

一切正常之后，清理测试文件： 

rm -v dummy.c a.out

注意

下一节中编制 Binutils 相当与额外检查了一次工具链是否正常编制。如果 Binutils 编制失败，就说明之前的 Binutils、GCC 或 Glibc 安装出现了问题。

有关此程序包的细节位于 第 6.9.4 节 “Glibc 的内容”。
